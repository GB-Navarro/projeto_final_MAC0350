```gherkin
Funcionalidade: Modelos Usuario, Aluno e Administrador
Regra de negócio: Usuario usa email como identificador único; Aluno e Administrador são perfis 1:1 vinculados a um Usuario.

  Cenário: criar Usuario com email como login
    Dado nenhum usuario cadastrado
    Quando um Usuario é criado com email "ana@exemplo.com" e senha
    Então o Usuario existe no banco com USERNAME_FIELD = email
    E o campo username não existe

  Cenário: criar Aluno vinculado a um Usuario
    Dado um Usuario do tipo ALUNO
    Quando um Aluno é criado vinculado a esse Usuario com genero, serie, escola e tipo_escola
    Então o Aluno existe no banco com todos os campos obrigatórios preenchidos
    E professor_nome e professor_email ficam nulos por padrão

  Cenário: codigo do Aluno é gerado no formato correto
    Dado um Aluno recém-criado
    Então o campo codigo começa com "E" e tem 7 caracteres no total
    E é único entre todos os alunos

  Cenário: criar Administrador vinculado a um Usuario
    Dado um Usuario do tipo ADM
    Quando um Administrador é criado vinculado a esse Usuario
    Então o Administrador existe no banco com aprovado = False
    E aprovado_por e aprovado_em ficam nulos

  Cenário: migrações aplicam sem erro
    Dado o projeto recém-clonado
    Quando "python manage.py migrate" é executado
    Então o comando termina sem erro
```
